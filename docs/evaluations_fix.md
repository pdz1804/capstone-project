# OmniDocBench Parsing Evaluation Summary

## 1. Overview

This document summarizes the methodology and evaluation framework for document parsing strategies on PDF documents within the project. The framework focuses on **Component-level Metrics**, comparing the structure extracted by our pipeline against standard benchmark annotations (OmniDocBench) to ensure no critical formatting, reading order, or component loss occurs.

## 2. Evaluation Data (Ground Truth)

This section uses a **real external benchmark**: **OmniDocBench**.

**Primary benchmark sources**
- GitHub: https://github.com/opendatalab/OmniDocBench
- Hugging Face dataset: https://huggingface.co/datasets/opendatalab/OmniDocBench
- Paper: https://arxiv.org/abs/2412.07626

For consistency with the evaluation snapshot summarized in this document, we describe the benchmark here as a **1650-page PDF parsing benchmark** spanning multiple document types, layout patterns, and languages. The annotations include rich block-level and span-level categories, plus:

- **reading-order annotations**
- **text recognition annotations** for text regions
- **LaTeX annotations** for formulas
- **LaTeX + HTML annotations** for tables
- **page-level / text-level / table-level attributes**

Examples of GT categories include `title`, `text_block`, `table`, `table_caption`, `figure`, `figure_caption`, `equation_isolated`, `header`, `footer`, `text_span`, and `equation_inline`.

In this repo, the local evaluation material is:

- **Evaluation Scope Used In This Summary**: 1650 pages

## 4. Evaluation Metrics

Important distinction:

- **Official OmniDocBench end-to-end leaderboard metric** from the benchmark repo is:

$$
\text{Overall}_{official} =
\frac{(1-\text{Text Edit Distance}) \times 100 + \text{Table TEDS} + \text{Formula CDM}}{3}
$$

- **This project does not report that official leaderboard score directly in this summary.**
  Instead, `backend/src/evaluation/parsing_info_loss/metrics.py` computes a **local component-level score** on top of OmniDocBench annotations. That local score is what is summarized below.

### 4.1 Local Overall Score Used In This Repo

Let the enabled metric groups be:

$$
G=\{\text{text},\text{table},\text{read\_order},\text{section\_hierarchy},\text{figure\_captioning}\}
$$

with default weights:

$$
w=\{30,25,15,15,15\}
$$

Then the reported overall score is:

$$
\text{Overall}_{local} =
\frac{\sum_{g \in G_{applicable}} w_g \cdot s_g}{\sum_{g \in G_{applicable}} w_g}
$$

If a metric group is disabled or not applicable, the denominator is renormalized over the remaining groups.

**Example:** if `text = 0.50`, `table = 0.70`, `read_order = 0.90`, `figure_captioning` is N/A, and `section_hierarchy` is disabled, then:

$$
\text{Overall}_{local} = \frac{30 \cdot 0.50 + 25 \cdot 0.70 + 15 \cdot 0.90}{30+25+15}=0.657
$$

### 4.2 Text Score (Weight 30%)

The local text score is **token-multiset F1**, not the official benchmark's text edit distance.

Let:

- $P$ = multiset token precision
- $R$ = multiset token recall

$$
P = \frac{|T_{gt} \cap T_{pred}|_{multi}}{|T_{pred}|}, \qquad
R = \frac{|T_{gt} \cap T_{pred}|_{multi}}{|T_{gt}|}
$$

$$
F1 = \frac{2PR}{P+R}
$$

The repo reports:

$$
\text{TextScore} = F1
$$

**Example:** if the GT has 100 normalized tokens, the prediction has 90 tokens, and 72 tokens overlap under multiset counting:

$$
P=72/90=0.80,\quad R=72/100=0.72,\quad F1=0.758
$$

### 4.3 Table Score (Weight 25%)

Each matched table is scored by combining official OmniDocBench **TEDS** with a text-based edit similarity:

$$
\text{TableContent}_j = 0.7 \cdot \text{TEDS}_j + 0.3 \cdot \text{EditSim}_j
$$

Table coverage is computed as an F1 over matched table instances:

$$
\text{CoverageF1} = \frac{2 \cdot \text{Precision}_{match} \cdot \text{Recall}_{match}}
{\text{Precision}_{match} + \text{Recall}_{match}}
$$

where:

$$
\text{Precision}_{match}=\frac{\#matched}{\#pred\_tables}, \qquad
\text{Recall}_{match}=\frac{\#matched}{\#gt\_tables}
$$

The final group score is:

$$
\text{TableScore} = 0.5 \cdot \text{CoverageF1} + 0.5 \cdot \operatorname{mean}_j(\text{TableContent}_j)
$$

**Example:** if matched-table coverage F1 is `0.80` and the mean content score across matched tables is `0.72`, then:

$$
\text{TableScore}=0.5 \cdot 0.80 + 0.5 \cdot 0.72 = 0.76
$$

### 4.4 Read Order Score (Weight 15%)

After greedily matching GT and predicted components, the evaluator compares all matched pairs `(a,b)` and checks whether their relative order is preserved:

$$
\text{ReadOrderScore} = \frac{\#correct\_pairs}{\#all\_pairs}
$$

**Example:** if 10 matched component-pairs are comparable and 9 preserve the same ordering as GT, then:

$$
\text{ReadOrderScore}=9/10=0.90
$$

### 4.5 Section Hierarchy Score (Weight 15%)

For section nodes, the evaluator combines:

$$
\text{NodeRecall}=\frac{\#matched\_sections}{\#gt\_sections}
$$

and per-node path accuracy:

$$
\text{PathNodeScore}_j = 0.7 \cdot \text{PathSim}_j + 0.3 \cdot \text{LevelMatch}_j
$$

$$
\text{PathAccuracy} = \operatorname{mean}_j(\text{PathNodeScore}_j)
$$

Final score:

$$
\text{SectionHierarchyScore}=0.6 \cdot \text{NodeRecall} + 0.4 \cdot \text{PathAccuracy}
$$

**Example:** if `NodeRecall = 0.75` and `PathAccuracy = 0.90`, then:

$$
\text{SectionHierarchyScore}=0.6 \cdot 0.75 + 0.4 \cdot 0.90 = 0.81
$$

# Retriever Evaluation Summary (Enhanced with Full Metrics)

## 1. Overview
This section summarizes the methodology and evaluation framework for retrieval strategies within the project. It focuses on whether the system can retrieve relevant text chunks and image/page evidence from the user's processed knowledge base before answer generation is considered.

Retrieval evaluation measures how well different retrieval strategies find relevant documents, chunks, and visual evidence using standard IR metrics such as Recall@K and nDCG@K.

## 2. Evaluation Data (Ground Truth)

This section uses an **internal synthetic retrieval benchmark** generated from active local documents, then stores the evaluation set in standard IR-style format (`queries` + `qrels`).

For the run summarized below:

- **Documents**: `2` active documents
- **Document groups**: `2412.19437v2_1` and `DeepSeek_V4`
- **Files represented**: `14` total files (`7 + 7`)
- **Questions generated**: `40`
- **Question categories**: `simple`, `complex_intent`, `reasoning`, `cross_file_reasoning`
- **Per-category count**: `10` questions each

These numbers should be read as **internal benchmark results over the project corpus**, not as leaderboard numbers on an external public benchmark.

## 3. Evaluated Strategies

The project implements multiple retrieval strategies:

1. **`SimpleBM25Retriever` (Sparse Retrieval)**: BM25Okapi algorithm, keyword-based, fast and efficient
2. **`SimpleDenseRetriever` (Dense Retrieval)**: Sentence transformers with FAISS, semantic understanding
3. **`SimpleHybridRetriever` (Hybrid Retrieval)**: Reciprocal Rank Fusion combining BM25 + Dense
4. **`ColQwenRetriever` (Image Retrieval)**: Vision-language model for PDF page semantic search

## 4. Evaluation Metrics

### 4.1 Core Metrics

- **Recall@K**: Proportion of relevant documents found in top K results
- **nDCG@K (Normalized Discounted Cumulative Gain)**: Ranking quality considering both relevance and position

Each metric is reported as:
- **Mean**: Arithmetic average across all 40 queries
- **Std Dev**: Standard deviation showing consistency across queries
